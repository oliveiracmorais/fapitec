# Story 1.2: Convenções Arquiteturais e Linguagem Ubíqua

## Status

Done

## Story

**Como** equipe responsável pela evolução da plataforma da `FAPITEC-SE`,  
**quero** formalizar convenções arquiteturais e de linguagem ubíqua do projeto,  
**para que** o código nasça consistente com `DDD`, `Clean Architecture`, `TDD` e com o vocabulário real do domínio.

## Acceptance Criteria

1. O projeto deve registrar explicitamente que a linguagem ubíqua do domínio será escrita em português.
2. O projeto deve definir o uso de padrões brasileiros para moeda, percentuais, datas, horários e mensagens.
3. O projeto deve registrar que valores monetários em `Go` não podem usar `float32` ou `float64`.
4. O projeto deve registrar `DDD`, `Clean Architecture`, `Clean Code` e `TDD` como princípios de implementação.
5. O projeto deve registrar `PostgreSQL + sqlc` como diretriz inicial para persistência no backend Go.
6. O projeto deve indicar que decisões de infraestrutura e entregas precisam respeitar as restrições de hospedagem e cronograma mensal do Termo de Referência.
7. A story deve manter checklist e file list atualizados.

## Convenções Consolidadas

As convenções oficiais para implementação inicial estão registradas em [docs/architecture/convencoes-arquiteturais-e-linguagem-ubiqua.md](/home/carlos/projetos/fapitec/docs/architecture/convencoes-arquiteturais-e-linguagem-ubiqua.md).

Síntese normativa:

- o domínio deve usar linguagem ubíqua em português;
- regras de negócio não devem depender diretamente de framework, banco, HTTP ou UI;
- o backend deve favorecer `DDD`, `Clean Architecture`, `Clean Code` e `TDD`;
- `PostgreSQL + sqlc` é a diretriz inicial de persistência;
- valores monetários em `Go` não podem usar `float32` ou `float64`;
- moeda, percentuais, datas, horários e mensagens devem seguir padrões brasileiros;
- decisões de infraestrutura e planejamento devem respeitar hospedagem e cronograma mensal do Termo de Referência;
- `Casdoor` e `Casbin` seguem em avaliação via ADR, sem acoplamento direto ao domínio.
