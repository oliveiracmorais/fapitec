# Story 1.18: Integração Casdoor como Plataforma de IAM

## Status

Concluida

## Story

**Como** equipe da `FAPITEC-SE`,
**quero** substituir a autenticação própria mínima pelo Casdoor como plataforma única de IAM,
**para que** o sistema tenha autenticação segura (OIDC/JWT), autorização RBAC gerenciável via UI e auditoria nativa, alinhado à decisão do ADR-001.

## Contexto

O ADR-001 foi finalizado em **Casdoor-only** (2026-06-01). A avaliação técnica confirmou que Casdoor já inclui Casbin internamente, cobrindo autenticação + autorização + auditoria em um único serviço. Atualmente o sistema usa:
- Autenticação própria mínima (bcrypt + sessão localStorage)
- Autorização inexistente (todos os endpoints públicos)
- Auditoria mínima (eventos registrados no banco local)

Esta story implementa a migração para Casdoor conforme previsto no ADR-001.

## Acceptance Criteria

1. Casdoor deve ser implantado via Docker Compose em **ambos os ambientes**:
   - **Local (desenvolvimento)**: como serviço no `apps/api/docker-compose.yml` ou `docker-compose.dev.yml` separado, acessível em `http://localhost:8000`
   - **Homologação**: no `docker-compose.homolog.yml` (Oracle Cloud VM ARM64), acessível via subdomínio `auth.fapitec.duckdns.org`
   Ambos com PostgreSQL dedicado.
2. O backend Go deve validar tokens JWT emitidos pelo Casdoor em todos os endpoints protegidos.
3. O middleware de autenticação deve extrair o usuário do token JWT e disponibilizá-lo no contexto da request.
4. O middleware de autorização deve consultar o Casdoor Enforce API para decidir acesso por perfil + módulo + operação.
5. Os 6 perfis institucionais (Story 1.4) devem ser cadastrados como roles no Casdoor: `administrador_fapitec`, `instituicao_ensino`, `funcionario_fapitec`, `diretoria`, `proponente`, `avaliador`.
6. A UI de gestão do Casdoor deve estar acessível:
   - **Local**: `http://localhost:8000` (sem SSL, admin/123)
   - **Homologação**: subdomínio com SSL (ex: `auth.fapitec.duckdns.org`)
7. O fluxo de login deve redirecionar para o Casdoor OIDC e retornar com JWT válido.
8. O fluxo de cadastro deve criar usuário no Casdoor via SDK Go mantendo as validações de CPF/passaporte no domínio.
9. A sessão deve ser mantida via JWT (httpOnly cookie ou Authorization header) — não mais via localStorage.
10. A auth própria atual deve ser mantida como fallback, ativável por feature flag (`AUTH_PROVIDER=casdoor|internal`).
11. Eventos de autorização (acesso negado, alteração de perfil, operação crítica) devem ser registrados via Casdoor Record API e consultáveis na UI de gestão do Casdoor.
12. O frontend Next.js deve consumir o fluxo OIDC sem expor client secret (proxy via backend Go ou server-side).
13. Testes de integração devem cobrir o adapter Casdoor.
