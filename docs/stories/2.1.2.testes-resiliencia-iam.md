# Story 2.1.2: Testes de Resiliência do Sistema IAM

## Status

Ready for Review

## Story

**Como** equipe de desenvolvimento da FAPITEC-SE,
**quero** executar uma bateria de testes de resiliência no sistema IAM (Casdoor + fallback internal),
**para que** o sistema se mantenha disponível e seguro mesmo em cenários de ataque, falha de provedor ou migração de configuração.

## Contexto

Referente à **Epic 2.1** (Finalização da Homologação e Pipeline CI/CD). O Casdoor IAM está integrado (Story 1.18) com feature flag `AUTH_PROVIDER` que permite alternar entre `casdoor` e `internal`. Esta story valida a resiliência do sistema: proteção contra força bruta, rollback do provedor de autenticação, fallback entre provedores, expiração de JWT e desabilitação de endpoints antigos.

## Acceptance Criteria

1. A proteção contra força bruta do Casdoor deve bloquear após 5 tentativas de login falhas por 15 minutos (configurável).
2. O rollback de `AUTH_PROVIDER=casdoor` para `internal` deve restaurar o fluxo de login/cadastro/sessão sem perda de dados.
3. A alternância entre provedores (`casdoor` ↔ `internal`) deve ser graciosa: usuários autenticados não perdem sessão ativa durante a transição.
4. Tokens JWT expirados devem ser rejeitados pelo middleware com status 401.
5. Tokens JWT com assinatura inválida devem ser rejeitados pelo middleware com status 401.
6. Endpoints antigos de autenticação (cadastro, login, redefinição de senha) devem retornar 410 Gone quando `AUTH_PROVIDER=casdoor`.
7. A auditoria de eventos de autenticação deve continuar funcionando em ambos os provedores.
8. A criação de novo usuário via Casdoor deve funcionar e o usuário deve conseguir fazer login OIDC.
9. A desabilitação de um usuário no Casdoor deve impedir login no sistema.
10. Resultados dos testes devem ser documentados em `docs/qa/testes-resiliencia-iam.md`.
