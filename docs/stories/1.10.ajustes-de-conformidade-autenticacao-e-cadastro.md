# Story 1.10: Ajustes de Conformidade em Autenticação e Cadastro

## Status

Ready for Review

## Story

**Como** equipe de desenvolvimento,
**quero** ajustar a implementação de autenticação e cadastro para atender integralmente aos requisitos do Termo de Referência e Projeto FAPITEC,
**para que** o sistema esteja em conformidade com as especificações contratuais antes da apresentação institucional.

## Acceptance Criteria

1. O cadastro deve aceitar CPF (brasileiros) ou Passaporte (estrangeiros), com validação específica para cada formato.
2. O DTO de cadastro deve incluir campo de confirmação de e-mail, validando que e-mail e confirmação são iguais.
3. As mensagens de erro devem seguir exatamente as especificações do Projeto FAPITEC.
4. O backend deve expor endpoints `GET /check-email` e `GET /check-cpf` para verificação de duplicidade.
5. O campo Nome deve ser validado: apenas letras e espaços, mínimo 3 caracteres.
6. A implementação deve preservar compatibilidade com a Story 1.9 (casos de uso existentes).
7. A story deve manter checklist e file list atualizados.

- [x] Validar confirmação de e-mail no caso de uso `CadastrarUsuario`

- [x] Atualizar testes existentes para refletir novas mensagens de erro

- Fonte primária: `docs/ProjetoFapitec.txt` (seções 1.1, 1.3 e Cadastro inicial)
- Story antecessora: `docs/stories/1.9.implementacao-autenticacao-e-cadastro.md`

- Passaporte: formato alfanumérico, sem validação de dígitos verificadores (diferente de CPF)
- As mensagens de erro devem ser exatamente:

  | Situação | Mensagem Esperada |
  |----------|------------------|
  | CPF/Passaporte ou senha inválidos | "CPF/Passaporte ou Senha inválidos. Tente novamente." |
  | CPF já cadastrado | "CPF já cadastrado no sistema." |
  | E-mail já cadastrado | "E-mail já cadastrado. Utilize outro endereço ou recupere sua senha." |
  | E-mail e confirmação diferentes | "O e-mail deve ser IGUAL ao e-mail principal." |
  | Senha e confirmação diferentes | "A senha deve ser IGUAL à primeira senha fornecida." |
