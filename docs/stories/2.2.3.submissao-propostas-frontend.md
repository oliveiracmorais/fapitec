# Story 2.2.3: Submissão de Propostas — Frontend

## Status

Done

## Story

**Como** proponente da FAPITEC-SE,
**quero** submeter propostas de pesquisa com um formulário multi-etapas,
**para que** eu possa concorrer aos recursos de fomento dos editais vigentes.

## Context

O backend de submissão de propostas (Story 2.2.2) já está concluído com endpoints REST, entidades de domínio e validações. O frontend atual possui páginas de editais (listagem, criação, detalhe) como referência de padrão. O domínio `inscricao_e_selecao_de_projetos` no backend expõe: `POST /api/v1/propostas`, `GET /api/v1/propostas`, `PUT /api/v1/propostas/{id}`, `DELETE /api/v1/propostas/{id}`, `POST /api/v1/propostas/{id}/submeter`.

O ProjetoFapitec.txt (Seção 5.8.2) especifica: interface responsiva, upload de documentos, edição dentro do prazo, geração de protocolo, histórico de versões, cálculo automático de orçamento.

## Acceptance Criteria

1. O proponente deve acessar um formulário multi-etapas ao clicar "Inscrever-se" em um edital ativo.
2. O formulário deve conter ao menos 4 etapas sequenciais: Dados do Proponente, Formação Acadêmica, Documentação, Planilha Orçamentária.
3. Dados do Proponente: nome, CPF, RG, gênero, etnia, data de nascimento, endereço, telefone — pré-preenchidos da sessão.
4. Formação Acadêmica: maior titulação, curso, instituição, ano de conclusão, área de conhecimento.
5. Documentação: upload de arquivos com tipos exigidos pelo edital (PDF, DOC, imagens), com validação de formato e tamanho.
6. Planilha Orçamentária: adicionar itens (descrição, tipo consumo/permanente, quantidade, valor unitário), cálculo automático do total.
7. O sistema deve exibir o valor total calculado e validar se excede o valor global do edital.
8. Ao submeter, o sistema deve exibir o protocolo gerado e redirecionar para o painel de acompanhamento.
9. O proponente deve poder editar a proposta enquanto o edital estiver vigente (botão "Editar" no painel).
10. O painel do proponente deve listar todas as suas propostas com status, protocolo, edital e ações disponíveis.
11. Os valores monetários devem usar formatação BRL (R$ 1.234,56).
12. A implementação deve seguir o mesmo padrão de componentes das páginas de editais existentes.

## Dev Notes

- **Padrão existente:** seguir o mesmo padrão das páginas de editais: `"use client"`, `useState` + `useEffect`, `fetch()` direto, Tailwind CSS, `CampoFormulario` para inputs
- **Upload:** usar `<input type="file">` nativo com validação client-side; backend espera FormData ou base64 (definir em conjunto com o adapter de upload)
- **Orçamento:** usar `Intl.NumberFormat('pt-BR', { style: 'currency', currency: 'BRL' })` para exibição; armazenar em centavos (int) no formulário
- **Etapas:** controlar etapa atual com `useState<number>`; validar etapa antes de avançar; permitir voltar
- **Rascunho:** criar proposta como rascunho ao iniciar o formulário (POST) para evitar perda de dados
- **Documentos:** nesta story, o upload será via endpoint separado a ser definido ou embutido no JSON como base64 (simplificação inicial)
- **Pré-preenchimento:** dados do proponente vêm da sessão (`useAuth().usuario`)
- **Middleware:** path `/minhas-propostas` já deve ser protegido pelo middleware existente (padrão `/(app)/*`)
- **Sem shadcn/ui:** o projeto não utiliza shadcn/ui; manter Tailwind puro + componentes inline
