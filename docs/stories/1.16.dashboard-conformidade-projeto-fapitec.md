# Story 1.16: Dashboard — Conformidade com ProjetoFapitec.txt e Termo de Referência

## Status

Ready for Review
✅ Todas as tasks implementadas e validadas (lint, typecheck, go vet)

## Story

**Como** equipe de desenvolvimento da FAPITEC-SE,
**quero** adequar o dashboard às especificações do `docs/ProjetoFapitec.txt` (RF 1.1 e 1.2) e `docs/TermoReferencia.txt` (módulos seção 4),
**para que** a tela inicial exiba indicadores, gráficos e cards de navegação para os módulos do sistema conforme o Termo de Referência.

> O `docs/TermoReferencia.txt` é o documento oficial e prevalece em caso de conflito com o `docs/ProjetoFapitec.txt`.

## Acceptance Criteria

1. O dashboard deve exibir cards de indicadores: Progresso ISO 27001 (gráfico de pizza, tom violeta), Não Conformidades Tratadas (gráfico de pizza, tom verde), Valor Absoluto (gráfico de linha), Status do Gráfico (barra de progresso, tom laranja).
2. Devem existir gráficos detalhados abaixo dos indicadores: um donut chart com 3 faixas percentuais e um gráfico de barra/linha.
3. O dashboard deve exibir cards de navegação para os módulos definidos no Termo de Referência (seção 4): Gestão de Editais, Inscrição e Seleção de Propostas, Contratação e Concessão de Apoios, Financeiro e Pagamentos, Prestação de Contas, Tomada de Contas Especial, Comunicação Institucional, Relatórios e Painéis de Indicadores, Gestão Documental. Cada card deve ser clicável e redirecionar para o módulo correspondente.
4. O card do usuário deve conter opção "Atualizar Perfil".
5. O backend deve expor `GET /api/v1/dashboard/indicadores` e `GET /api/v1/dashboard/graficos`.
6. Todos os botões, cards de módulo e gráficos devem ter `aria-label` (acessibilidade).
7. Gráficos e cards devem ter tooltips explicativos ao passar o cursor.
8. Design responsivo (cards se reorganizam verticalmente em telas menores).

### Code Review Findings

#### ✅ Positivos

- **Acessibilidade**: Todos os elementos interativos têm `aria-label`, `role` e suporte a teclado (Enter/Space nos cards módulo)
- **Gráficos nativos**: MiniPieChart SVG sem dependência externa; MiniLineChart com recharts ResponsiveContainer
- **Responsividade**: Grid Tailwind adaptável; gráficos com ResponsiveContainer ajustam largura automaticamente
- **Tooltips**: `title` attr nos cards + recharts Tooltip nos gráficos
- **Botão Atualizar Perfil**: Visível, com aria-label e rota /perfil

#### ⚠️ Observações

1. **`router.push` type cast** (`page.tsx:255`): Uso de `(router.push as (u: string) => void)(url)` para contornar tipagem — funcional, mas suprime o retorno `Promise<boolean>`. Não crítico, mas idealmente usar `void router.push(url)` ou `router.push(url).catch(() => {})`.

2. **Dados mockados**: Ambos endpoints retornam dados mockados inline. OK para esta fase, mas precisa de migration para dados reais quando houver módulo de negócio implementado.

3. **`formatarValor` só usado para indicador linha**: Função existe mas só chamada em `ind.tipo === "linha"` — não usado para cards do tipo numero (não há nenhum indicador com tipo "numero" atualmente).

4. **Sem testes automatizados para dashboard**: Nenhum test file para os handlers do dashboard (`cmd/api` sem test files). Os dados mockados são triviais, mas quando evoluir para dados reais, testes de integração serão necessários.

### Risk Assessment

| Risco | Probabilidade | Impacto | Mitigação |
|-------|:-----------:|:-------:|-----------|
| Rotas dos módulos levam a 404 | Alta | Médio | Já identificado no handoff do dev — páginas serão criadas em stories futuras |
| Dados mockados estagnados | Média | Baixo | Feature flag ou fallback visual quando endpoint retornar erro (já implementado: `.catch(() => [])`) |
| router.push sem await | Baixa | Baixo | Funcional, apenas warning de lint suprimido |
