# Story 2.2.4: Gestão de Avaliadores e Atribuição

## Status

Concluded

## Story

**Como** administrador da FAPITEC-SE,
**quero** cadastrar avaliadores, atribuí-los a editais e gerenciar convites,
**para que** as propostas submetidas sejam avaliadas por profissionais qualificados com garantia de anonimato e prevenção de conflito de interesses.

## Context

O seed de perfis já inclui o perfil `avaliador` com permissões básicas (`editais:visualizar`, `dashboard:visualizar`). O domínio `inscricao_e_selecao_de_projetos` possui entidades `Proposta` e `Parecer` (com `AvaliadorID`), mas não existe entidade `Avaliador` com dados de perfil, vínculo com usuário, área de conhecimento, titulação e estado.

O fluxo de avaliação completo (Story 2.2.5) depende do cadastro e atribuição de avaliadores. O backend precisa expor:
- Cadastro de perfil avaliador (CRUD)
- Atribuição a editais com período de atuação
- Convite e aceite (mudança de status)
- Mecanismo de anonimização (IDs opacos)

O Termo de Referência (Seção 4.2) e o ProjetoFapitec.txt especificam necessidade de: cadastro de consultores ad hoc, área de especialização, evitamento de conflito de interesses, e anonimato durante avaliação.

## Acceptance Criteria

1. O administrador deve poder cadastrar um avaliador com: nome, CPF, e-mail, titulação máxima, área de conhecimento, instituição, currículo resumido e estado (ativo/inativo).
2. O vínculo do avaliador deve ser com um usuário existente no sistema (via `usuario_id`), validando que o perfil do usuário é `avaliador`.
3. O administrador deve poder atribuir avaliadores a editais específicos com data de início e fim de atuação.
4. O avaliador deve receber um convite ao ser atribuído a um edital e poder aceitar ou recusar.
5. O sistema deve gerar um identificador opaco (hash) para cada avaliador em cada edital, garantindo anonimato durante a avaliação.
6. O administrador deve poder listar avaliadores com busca por nome, CPF e área de conhecimento.
7. O administrador deve poder visualizar a carga de trabalho do avaliador (quantos editais ativos, quantas propostas para avaliar).
8. O frontend deve ter uma página de listagem de avaliadores com tabela, busca e filtros.
9. O frontend deve ter uma página de cadastro/edição de avaliador com formulário.
10. O frontend deve ter uma seção de atribuição a editais dentro do formulário de avaliador.

## Dev Notes

- **Entidade `Avaliador` vs `UsuarioSessao`:** `Avaliador` é uma entidade de negócio separada do usuário de identidade. O `usuario_id` vincula ao `UsuarioSessao` do Casdoor. Validar que o perfil Casdoor do usuário é `avaliador`.
- **Hash de anonimização:** Gerar hash deterministicamente com `SHA-256(avaliador_id + ":" + edital_id + ":" + salt)`. O salt deve ser fixo por ambiente (configurável via env `AVALIADOR_HASH_SALT`). O hash é armazenado em `atribuicoes_editais.hash_anonimizacao`.
- **Atribuição:** O endpoint `POST /avaliadores/{id}/atribuir` recebe edital_id + datas. Cria registro com status "pendente". O convite pode ser aceito/recusado via `PUT /atribuicoes/{id}/convite`.
- **Perfil `avaliador` existente:** O seed já cria o perfil no Casdoor. Verificar se é necessário adicionar permissões específicas para o módulo de avaliação.
- **Autorização:** Apenas administradores (`perfil = administrador_fapitec`) podem cadastrar, editar, listar e atribuir avaliadores. O avaliador pode apenas visualizar suas atribuições e gerenciar convites.
- **Frontend:** Seguir padrão estabelecido: `"use client"`, `useState` + `useEffect`, `fetch()` direto, Tailwind CSS, componentes `CampoFormulario`.
- **Estado do Avaliador:** `ativo` = pode receber atribuições; `inativo` = não pode receber novas atribuições (atribuições existentes continuam ativas até o fim do período).
