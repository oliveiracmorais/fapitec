# Story 1.9: Implementação de Autenticação e Cadastro

## Status

Ready for Review

## Story

**Como** equipe de desenvolvimento da plataforma da `FAPITEC-SE`,
**quero** implementar a autenticação e cadastro inicial no backend Go com DDD e Clean Architecture,
**para que** usuários possam se cadastrar e autenticar com segurança, respeitando a fronteira arquitetural definida na Story 1.3.

## Acceptance Criteria

1. O backend Go deve implementar os casos de uso `CadastrarUsuario` e `AutenticarUsuario` no contexto `identidade_e_acesso`.
2. O dominio deve conter entidades `Usuario`, objetos de valor `CPF`, `Email` e `SenhaHash`, sem dependencia de framework.
3. O sistema deve validar CPF (algoritmo de digitos verificadores) e email (formato) no dominio.
4. A senha deve ser armazenada com hash seguro (bcrypt) via adaptador de infraestrutura.
5. O caso de uso `AutenticarUsuario` deve rejeitar apos 5 tentativas falhas consecutivas com bloqueio de 15 minutos.
6. Migracoes SQL devem criar a tabela `usuarios` com colunas para bloqueio (tentativas, bloqueado_ate).
7. O repositorio deve ser implementado com `sqlc` (consultas SQL explicitas, sem ORM).
8. A CLI deve expor comandos de autenticacao via `scripts/plataforma-shell.js` antes de qualquer interface grafica.
9. A implementacao deve registrar eventos de auditoria para login, falha de login, bloqueio e cadastro.
10. A implementacao deve seguir a estrutura DDD: `dominio/`, `aplicacao/`, `interfaces/`, `infraestrutura/`.
11. A story deve manter checklist e file list atualizados.
