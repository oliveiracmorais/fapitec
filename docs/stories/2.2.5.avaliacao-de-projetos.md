# Story 2.2.5: Avaliação de Projetos

## Status

Concluded

## Story

**Como** avaliador da FAPITEC-SE,
**quero** emitir parecer técnico sobre propostas submetidas a editais aos quais fui atribuído,
**para que** as propostas sejam classificadas com base em critérios objetivos e transparência.

## Context

As stories anteriores estabeleceram a base necessária para a avaliação:

- **Story 2.2.1**: Extensão do modelo de editais — inclui `nota_de_corte` e critérios de avaliação no JSONB
- **Story 2.2.2**: Submissão de propostas — entidade `Proposta` com campo `Pareceres []Parecer`, value object `StatusPropostaEmAvaliacao`
- **Story 2.2.3**: Frontend de submissão — painel do proponente
- **Story 2.2.4**: Gestão de avaliadores e atribuição — cadastro, atribuição a editais, hash de anonimização, convite

### Estado atual da infraestrutura de parecer

A tabela `pareceres` existe no banco (migration 005) com os campos: `id`, `proposta_id`, `etapa`, `avaliador_id`, `nota`, `parecer_texto`, `data`. A entidade `Parecer` e o `ParecerDTO` estão definidos. No entanto:

- **Não há caso de uso** para emitir/registrar parecer
- **Não há queries sqlc** para CRUD de pareceres (apenas `COUNT(*)` em `query_avaliadores.sql`)
- **Não há endpoints REST** para parecer
- **Não há método `AdicionarParecer`** na entidade `Proposta`
- **Não há transição de status** `submetida → em_avaliacao → aprovada/reprovada`
- **Não há cálculo de classificação/ranking** de propostas
- **Não há integração com `HashAnonimizacao`** para garantir anonimato na avaliação
- **Não há validação de nota** (mínimo/máximo por critério)
- **Não há notificação** ao avaliador sobre novas propostas para avaliar
- **Não há frontend** para painel do avaliador

O Termo de Referência (Seção 4.2) especifica: avaliação por consultores ad hoc, critérios definidos no edital, nota por critério, nota final calculada automaticamente, anonimato do avaliador, e divulgação de resultados.

## Acceptance Criteria

### Backend

1. O sistema deve permitir que um avaliador atribuído a um edital (com convite aceito) emita parecer para propostas daquele edital.
2. O parecer deve conter: `proposta_id`, `avaliador_id`, `etapa`, `nota` (0-100), `parecer_texto`, e `data`.
3. O sistema deve validar que o avaliador está ativo, tem convite aceito para o edital, e a proposta pertence ao edital.
4. Ao emitir o primeiro parecer, a proposta deve transicionar de `submetida` para `em_avaliacao`.
5. O sistema deve garantir o anonimato do avaliador: o hash de anonimização deve ser usado para identificar o avaliador nas visualizações de parecer pelo administrador.
6. O administrador deve poder listar pareceres de uma proposta (por etapa, avaliador anonimizado).
7. O avaliador deve poder listar as propostas pendentes de avaliação nos editais aos quais está atribuído.
8. O avaliador deve poder editar seu próprio parecer enquanto a proposta estiver em avaliação.
9. O sistema deve calcular a nota final da proposta (média simples ou ponderada, conforme configurado no edital).
10. O administrador deve poder finalizar a avaliação de um edital, gerando a classificação das propostas.

### Frontend

11. O avaliador deve ter um painel com as propostas pendentes de avaliação.
12. O avaliador deve ter um formulário de parecer com campos: nota (slider/input numérico), parecer textual.
13. O administrador deve ter uma tela de acompanhamento de avaliações com status por proposta e avaliador.
14. O administrador deve ter uma tela de classificação final das propostas.

## Dev Notes

- **Hash de anonimização:** Usar o `HashAnonimizacao` já implementado na Story 2.2.4 para anonimizar o avaliador nas listagens de parecer. O hash é `SHA-256(avaliador_id + ":" + edital_id + ":" + salt)`.
- **Nota:** Usar representação inteira (0-100), sem decimais. Se o edital exigir média ponderada, os pesos são definidos no JSONB `critérios_avaliacao` do edital.
- **Transição de status:** `submetida` → `em_avaliacao` (1º parecer), `em_avaliacao` → `aprovada`/`reprovada` (finalização pelo admin).
- **Autorização:** Apenas avaliadores com convite aceito e estado ativo podem emitir parecer. Apenas admin pode finalizar avaliação e ver classificação.
- **Nota de corte:** Definida no edital (`nota_de_corte`). Propostas com nota final >= corte são aprovadas.
- **Múltiplos avaliadores por proposta:** Cada avaliador emite parecer independente. A nota final é a média (ou média ponderada) dos pareceres.
