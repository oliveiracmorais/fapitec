# Story 1.7: Integração Runtime Mínima de Autenticação, Autorização e Auditoria

## Status

Ready for Review

## Story

**Como** equipe da `FAPITEC-SE`,  
**quero** validar um fluxo runtime mínimo de autenticação, autorização por perfis e auditoria no shell inicial,  
**para que** a apresentação inicial demonstre o alicerce de segurança e rastreabilidade antes do detalhamento dos módulos de negócio.

## Acceptance Criteria

1. O sistema deve oferecer um fluxo mínimo de entrada no shell, usando credenciais ou identidades demonstrativas controladas para validação, sem coletar dados reais de usuários da `FAPITEC-SE`.
2. O dashboard não deve depender de perfil fixo hardcoded; o perfil ativo deve vir de uma sessão ou contexto demonstrável de autenticação.
3. O acesso ao dashboard deve ser negado quando não houver sessão ativa.
4. A visibilidade dos cards e o acesso às páginas placeholder devem respeitar os perfis e permissões definidos na Story `1.4`.
5. Tentativas de acesso negado devem produzir evento de auditoria compatível com a Story `1.5`.
6. A navegação bem-sucedida para módulos pelo dashboard deve produzir evento de auditoria compatível com a Story `1.5`.
7. A solução deve preservar a premissa `CLI First`, mantendo ou ampliando comandos para simular sessão, verificar acesso e inspecionar eventos auditáveis.
8. O fluxo deve deixar explícito que a autenticação é uma base mínima de validação e não fecha a decisão arquitetural sobre `Casdoor` e `Casbin`.
9. O fluxo deve usar linguagem institucional em português e padrões brasileiros de data/hora na apresentação.
10. A story deve manter checklist e file list atualizados.

## Escopo Da Fatia

Esta story deve transformar o shell inicial da Story `1.6` em um fluxo demonstrável mais próximo do runtime esperado:

- entrada controlada por identidade demonstrativa;
- sessão mínima para a apresentação;
- autorização real aplicada ao dashboard e às páginas placeholder;
- eventos de auditoria gerados por ações de navegação;
- roteiro CLI e UI coerente para validação pela `FAPITEC-SE`.

Esta story não deve implementar cadastro real de usuários, recuperação real de senha, integração definitiva com provedor externo, MFA, gestão administrativa completa de perfis ou regras finas de autorização por etapa de processo.

## Regras De Demonstração

- Não usar dados pessoais reais.
- Não apresentar a autenticação demonstrativa como solução final de produção.
- Identidades demonstrativas devem representar perfis institucionais mínimos, como `administrador`, `gestor_fapitec`, `proponente`, `avaliador` e `auditoria_controle`.
- A navegação por perfil deve evidenciar pelo menos um caso de acesso permitido e um caso de acesso negado.
- Eventos de auditoria não devem registrar senhas, tokens, segredos ou dados sensíveis.

## Integração Esperada

- Autenticação: cria ou simula uma sessão mínima para a apresentação.
- Autorização: consome o catálogo de perfis e permissões para filtrar cards e bloquear rotas.
- Auditoria: registra ações e resultados com ator, perfil, ação, resultado, data/hora e contexto básico.
- UI: apresenta o shell e os módulos como placeholders de validação, sem detalhar regras internas dos módulos de negócio.
- CLI: continua sendo a primeira forma de validar módulos, perfis, acesso e auditoria.

- O domínio não deve depender diretamente de framework, banco, HTTP, UI, `Casdoor` ou `Casbin`. [Source: docs/architecture/convencoes-arquiteturais-e-linguagem-ubiqua.md#9-relação-com-identidade-e-acesso]
- `Casdoor` e `Casbin` seguem em avaliação; esta story não deve fechar essa decisão. [Source: docs/architecture/adr-001-identidade-e-acesso.md#decisão-em-avaliação]
