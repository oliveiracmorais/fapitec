# Story 2.1.3: Pipeline CI/CD para Deploy em Homologação

## Status

Concluído

## Story

**Como** equipe de infraestrutura da FAPITEC-SE,
**quero** criar uma pipeline automatizada via GitHub Actions que execute lint, typecheck, testes, build e deploy automático no ambiente de homologação ao fazer push na branch `main`,
**para que** o deploy seja padronizado, reproduzível e seguro, reduzindo risco operacional e acelerando entregas.

## Contexto

Referente à **Epic 2.1** (Finalização da Homologação e Pipeline CI/CD). Atualmente o deploy na Oracle Cloud VM é feito manualmente via script `scripts/deploy-homologacao.sh`. Esta story cria uma pipeline CI/CD completa com GitHub Actions, integrando validações de qualidade (lint, typecheck, testes), deploy via SSH e rollback automático em caso de falha.

## Acceptance Criteria

1. O pipeline deve ser acionado automaticamente ao fazer push na branch `main`.
2. O pipeline deve executar as etapas sequenciais: lint → typecheck → test → build → deploy.
3. A etapa de lint deve usar `npm run lint` e falhar o pipeline se houver erros.
4. A etapa de typecheck deve usar `npm run typecheck` e falhar o pipeline se houver erros.
5. A etapa de teste deve usar `npm test` (e `go test ./...` no backend) e falhar o pipeline se houver falhas.
6. A etapa de build deve gerar os artefatos Docker e enviá-los para a VM via SSH.
7. A etapa de deploy deve executar `scripts/deploy-homologacao.sh` via SSH na VM.
8. Após o deploy, deve haver um healthcheck HTTP validando que a aplicação responde com 200 em `/health` e `/api/v1/health`.
9. Se o healthcheck falhar, o pipeline deve executar rollback automático (restaurar versão anterior e reiniciar containers).
10. O pipeline deve notificar o resultado (sucesso/falha) em um canal de comunicação (Slack ou e-mail).
11. Secrets do GitHub (`SSH_HOST`, `SSH_KEY`, `DOCKER_COMPOSE_VARS`) devem estar configurados e referenciados no pipeline.
12. O pipeline não deve interferir em deploys manuais existentes.
